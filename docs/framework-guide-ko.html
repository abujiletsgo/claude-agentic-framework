3-Phase Pipeline

모든 요청은 분류, 실행, 학습의 3단계 파이프라인을 거칩니다. 각 단계에서 Hook이 자동으로 개입합니다.

분류 → 실행 → 학습
1
Phase 1: 분류 (Classification)
analyze_request.py (Caddy Classifier)가 요청의 복잡도, 유형, 품질, 범위를 4차원으로 분류합니다. auto_delegate.py가 분류 결과에 따라 적절한 전략(direct, team, orchestrate, rlm, fusion)으로 라우팅합니다.
2
Phase 2: 실행 (Execution)
Damage control이 위험 명령을 차단하고, Session lock이 동시성을 방지합니다. 에이전트가 작업을 수행하며, PostToolUse 훅이 오류 분석, 컨텍스트 관리, 코드 리뷰를 실시간으로 수행합니다.
3
Phase 3: 학습 (Learning)
auto_fact_extractor.py가 실행 결과를 FACTS.md에 기록합니다. auto_memory_writer.py가 세션 종료 시 MEMORY.md에 요약을 저장합니다. auto_skill_generator.py가 반복 패턴을 스킬로 자동 생성합니다.

4가지 Hook 유형

command 5-500ms

셸 스크립트, Python 스크립트, 또는 Rust 바이너리를 직접 실행합니다. stdin으로 JSON 입력, stdout으로 결과 반환. 19개의 훅은 caf-hooks Rust 바이너리로 실행되어 5-30ms (Python 대비 6-32x 빠름).

prompt ~0ms 생성

LLM 컨텍스트에 인라인으로 삽입됩니다. 외부 프로세스 없이 프롬프트 내에서 평가됩니다. 추가 지침이나 경고를 주입할 때 사용합니다.

http ~50ms

HTTP endpoint로 요청을 보냅니다. 외부 서비스 연동이나 웹훅에 사용됩니다. Observability 앱과의 통신에 활용됩니다.

agent ~2-5s

Sub-agent를 생성하여 검증 작업을 수행합니다. 가장 느리지만 가장 강력합니다. auto_review_team.py가 이 유형으로 위험 Bash 명령을 검토합니다.

Damage Control — 3-Tier Defense

allow: ["*"] 환경에서도 실수를 방지합니다. 3단계 방어로 위험한 명령을 차단합니다.

T1
Permission Rules — settings.json allow/deny/ask

Claude Code의 내장 권한 시스템. allow: ["*"]로 전체 허용하되, 프레임워크의 추가 방어 계층이 보호합니다. 패턴 매칭보다 먼저 평가됩니다.

T2
Pattern Matching — unified-damage-control.py + patterns.yaml

정규식 패턴으로 위험 명령을 차단합니다. rm -rf /, git push --force main, DROP TABLE 등을 차단. patterns.yaml에서 패턴 추가/수정. 안전한 명령 목록(safe_commands)으로 오탐 방지.

T3
Path Protection — 경로별 접근 제어

파일 시스템 수준의 보호. zero-access (접근 자체 차단), read-only (읽기만 허용), no-delete (삭제만 차단)의 3단계로 경로를 보호합니다. settings.json, .env, 인증서 파일 등이 보호 대상입니다.
