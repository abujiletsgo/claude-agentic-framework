3-Phase Pipeline

모든 요청은 분류, 실행, 학습의 3단계 파이프라인을 거칩니다. 각 단계에서 Hook이 자동으로 개입합니다.

분류 → 실행 → 학습
1
Phase 1: 분류 (Classification)
analyze_request.py (Caddy Classifier)가 요청의 복잡도, 유형, 품질, 범위를 4차원으로 분류합니다. auto_delegate.py가 분류 결과에 따라 적절한 전략(direct, team, orchestrate, rlm, fusion)으로 라우팅합니다.
2
Phase 2: 실행 (Execution)
Damage control이 위험 명령을 차단하고, Session lock이 동시성을 방지합니다. 에이전트가 작업을 수행하며, PostToolUse 훅이 오류 분석, 컨텍스트 관리, 코드 리뷰를 실시간으로 수행합니다.
3
Phase 3: 학습 (Learning)
auto_fact_extractor.py가 실행 결과를 FACTS.md에 기록합니다. auto_memory_writer.py가 세션 종료 시 MEMORY.md에 요약을 저장합니다. auto_skill_generator.py가 반복 패턴을 스킬로 자동 생성합니다.

4가지 Hook 유형

command 200-500ms

셸 스크립트 또는 Python 스크립트를 직접 실행합니다. stdin으로 JSON 입력, stdout으로 결과 반환. 대부분의 Hook이 이 유형입니다.

prompt ~0ms 생성

LLM 컨텍스트에 인라인으로 삽입됩니다. 외부 프로세스 없이 프롬프트 내에서 평가됩니다. 추가 지침이나 경고를 주입할 때 사용합니다.

http ~50ms

HTTP endpoint로 요청을 보냅니다. 외부 서비스 연동이나 웹훅에 사용됩니다. Observability 앱과의 통신에 활용됩니다.

agent ~2-5s

Sub-agent를 생성하여 검증 작업을 수행합니다. 가장 느리지만 가장 강력합니다. auto_review_team.py가 이 유형으로 위험 Bash 명령을 검토합니다.

Circuit Breaker

훅 실패가 전체 시스템을 중단시키지 않도록 보호합니다.

1
훅 실패 감지
훅이 비정상 종료 코드 반환 또는 타임아웃 (5-15초) 초과
2
Circuit Breaker 트립
~/.claude/circuit_breakers/에 브레이커 파일 생성 — 해당 훅 비활성화
3
자동 복구
60초 쿨다운 후 자동 복구. 수동: 브레이커 파일 삭제
4
나머지 훅 정상 작동
다른 훅은 영향 없이 계속 실행 — 격리된 장애 처리
예시: Damage Control이 위험 명령어를 차단하는 과정
T

!

1. Quoted content stripping: 따옴표/heredoc 내용 제거
2. Pattern matching: \bgit\s+push\s+--force\b → 매칭!
3. Reason: "Force push는 원격 기록을 덮어씁니다"
X
결과: BLOCKED
출력: {"hookSpecificOutput": {"permissionDecision": "deny", "permissionDecisionReason": "Force push는 원격 기록을 덮어씁니다"}}
Claude에게 차단 사유가 전달됨 → 대안 제시 (일반 push 또는 사용자 확인 요청)
예시: Circuit Breaker가 장애를 격리하는 과정
1
auto_cost_warnings.py 실패 (3회 연속)
외부 API 응답 지연으로 타임아웃 (15초 초과). 3회 연속 실패 감지.

60s

Agent System — 8 Agents

7개의 Opus 에이전트와 1개의 Haiku 에이전트. 각 에이전트는 특화된 역할, 도구 제한, 격리 수준을 가집니다.

역할: 메인 코디네이터 · initialPrompt: /prime
tools
Task, Read, Glob, Grep, Bash, Write, Edit
역할
복잡한 multi-agent 작업을 분해하고 조율. TaskList로 병렬 실행 관리.
왜 필요한가
복잡한 조율에는 최고 수준의 추론이 필요합니다.

역할: 아키텍처 감독 · isolation: worktree

Damage Control — 3-Tier Defense

allow: ["*"] 환경에서도 실수를 방지합니다. 3단계 방어로 위험한 명령을 차단합니다.

T1
Permission Rules — settings.json allow/deny/ask

Claude Code의 내장 권한 시스템. allow: ["*"]로 전체 허용하되, 프레임워크의 추가 방어 계층이 보호합니다. 패턴 매칭보다 먼저 평가됩니다.

T2
Pattern Matching — unified-damage-control.py + patterns.yaml

정규식 패턴으로 위험 명령을 차단합니다. rm -rf /, git push --force main, DROP TABLE 등을 차단. patterns.yaml에서 패턴 추가/수정. 안전한 명령 목록(safe_commands)으로 오탐 방지.

T3
Path Protection — 경로별 접근 제어

파일 시스템 수준의 보호. zero-access (접근 자체 차단), read-only (읽기만 허용), no-delete (삭제만 차단)의 3단계로 경로를 보호합니다. settings.json, .env, 인증서 파일 등이 보호 대상입니다.

Architecture Map

/arch-map 명령이 생성하는 5가지 산출물입니다.

1
Blast-Radius Table
"X 변경 시 Y 업데이트"
  • 모든 파일 간 의존성을 매핑합니다. 하나의 파일을 변경했을 때 영향을 받는 모든 파일을 테이블로 표시합니다.
2
Mermaid Diagram
의존성 시각화
  • Mermaid 포맷의 의존성 다이어그램 생성. 모듈 간 관계를 시각적으로 표현합니다.
3
Critical Paths
핵심 워크플로우
  • 시스템의 핵심 경로를 식별합니다. SessionStart → PreToolUse → Stop 같은 중요 흐름을 추적합니다.
4
Data Lineage
데이터 파일 계보
  • 데이터 파일이 어디서 생성되고 어디서 소비되는지 추적합니다. YAML, JSON, TOML 설정 파일의 흐름을 시각화합니다.
5
Duplication Warnings
중복 경고
  • 코드 또는 설정의 중복을 감지하고 경고합니다. DRY 원칙 위반을 자동으로 식별합니다.
저장 위치: .claude/ARCHITECTURE.md에 저장되며 프로젝트 컨텍스트와 통합됩니다.
