What are Skills?

Skills vs Agents

Skills and agents solve different problems. Knowing which to use prevents over-engineering.

Skills /slash-command

A prompt template that executes in the current Claude session. No separate context, no agent spawning. Think of it as a "macro" that loads a specific set of instructions and executes.

  • ✓ Runs inline — no context overhead
  • ✓ Has access to current conversation history
  • ✓ Invoked with /skill-name
  • ✓ Best for: focused tasks with a known workflow

Agents Task tool

A subprocess with its own isolated context. Spawned via the Task tool. Runs independently, returns a result. The parent doesn't accumulate the agent's exploration in context.

  • ✓ Isolated context — no context rot
  • ✓ Can run in parallel with other agents
  • ✓ Best for: heavy exploration, parallel work
  • ✓ Returns findings, not file contents
Rule of thumb: Use a skill when you know exactly what needs to happen. Use an agent when you need to explore first.
Framework Skills

The 7 Global Skills

Installed at ~/.claude/skills/ by bash install.sh. Available in every project.

code-review
/code-review [file or PR]
PR-level code review. Checks for bugs, security issues, performance problems, and style violations. Returns structured findings by severity: critical → major → minor → suggestion.
Examples
/code-review src/auth/session.ts
Checks: SQL injection, insecure JWT config, missing input validation, race conditions
/code-review — review the last 3 changed files
error-analyzer
/error-analyzer [paste error or describe]
Root cause analysis for errors, exceptions, and crashes. Reads the stack trace, identifies the origin, and suggests a concrete fix. Cross-references FACTS.md GOTCHAS automatically.
Example
/error-analyzer
Then paste the stack trace → returns: root cause, affected file:line, suggested fix, how to verify
facts new
/facts [subcommand]
Manages your project's FACTS.md — the living fact sheet that prevents hallucination. Subcommands: list, add CATEGORY "text", stale "pattern", summary, purge-stale.
Common uses
/facts list
Shows all facts by category with dates and authors
/facts add CONFIRMED "deploy command is: npm run deploy:prod"
Adds verified fact with today's date + your git username
/facts stale "webpack"
Moves any webpack-related fact to STALE (e.g., after migrating to Vite)
Why this matters: Facts only get deleted from the STALE junk pile (after 90 days). Your CONFIRMED/GOTCHAS/PATHS facts persist indefinitely — a project working fine for 2 years keeps all its knowledge.
refactoring-assistant
/refactoring-assistant [target]
Safe, incremental refactoring. Analyzes the target, identifies what can change, proposes a sequence of small safe steps, and executes with testing between each step.
Example
/refactoring-assistant UserService class
Step 1: extract interface → Step 2: split into UserService + UserRepository → Step 3: update callers → validates each step
security-scanner
/security-scanner [path or scope]
Vulnerability audit against OWASP Top 10 and common patterns. Checks: injection, broken auth, XSS, insecure deserialization, misconfiguration. Returns findings with severity and remediation.
Example
/security-scanner src/api/
CRITICAL: SQL injection in search endpoint (line 47) — unsanitized user input in raw query
MAJOR: JWT secret hardcoded in config.ts — move to environment variable
test-generator
/test-generator [file or function]
Generates comprehensive test suites. Reads the target code, identifies edge cases, error paths, and happy paths. Writes tests in your project's existing test framework (Jest, Vitest, pytest, etc.).
Example
/test-generator src/utils/price.ts
Generates: unit tests for formatPrice(), edge cases (null, negative, overflow), locale tests, snapshot for currency display
knowledge-db
/knowledge-db [subcommand]
SQLite FTS5 knowledge database for cross-session learnings. Store and retrieve decisions, patterns, and solutions that span multiple projects. Separate from FACTS.md — this is for cross-project knowledge.
Example
/knowledge-db search "prisma connection pooling"
Returns: past learnings about Prisma setup from other projects you worked on
/knowledge-db store "always set connectionLimit: 5 in Prisma for Lambda"
Global vs Local

Layered Skill Architecture

Global skills work everywhere. Local skills are project-specific — and they win over globals with the same name.

Global Skills

~/.claude/skills/ (symlinked by install.sh)

7 framework skills. Available in all projects. Invoked with their name: /code-review, /facts, etc.

Local Skills

.claude/skills/ (project directory)

Project-specific. Overrides global with same name. A local code-review skill adds project-specific rules on top of the global one.

Creating a Local Skill

# .claude/skills/deploy.md
---
name: deploy
description: >
  Deploy this application to production. Handles environment-specific
  config, database migrations, and health checks.
---

# Deploy

Run the production deployment sequence for this project:

1. Run tests: `npm run test:ci`
2. Build: `npm run build`
3. Run migrations: `npx prisma migrate deploy`
4. Deploy to Fly.io: `fly deploy --config fly.production.toml`
5. Verify health: `curl https://api.ourapp.com/health`

If any step fails, stop and report the error. Do not proceed to the next step.

After successful deploy, add a CONFIRMED fact: "Last deploy: [date]"
Invoke with /deploy — Claude loads the skill and executes the exact sequence above. No memorizing commands, no config hunting. The skill is the runbook.
Advanced

Skills for Team Workflows

Local skills are the best way to encode team conventions. Everyone on the team gets the same workflow by default.

PR checklist skill

# .claude/skills/pr-check.md
Before creating a PR, verify:
- Tests pass: npm test
- No console.log statements in src/
- API changes documented in docs/api.md
- Migration has a down() function
- New env vars added to env.example

Environment setup skill

# .claude/skills/setup.md
Set up local development:
1. Copy env.example to env file
2. docker compose up -d postgres redis
3. npm install
4. npx prisma migrate dev
5. npm run seed
6. npm run dev (port 3000)
