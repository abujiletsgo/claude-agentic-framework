Data Flow

Session Start — What Auto-Fires

Lean by design: only project architecture is injected automatically. Two hooks, minimal overhead.

SessionStart hooks → context built before first message
🌐
~/.claude/CLAUDE.md + MEMORY.md L3 · always ~800 tok
Global behavioral rules and cross-project learnings. Always present. Cannot be overridden by project config. Highest-priority behavioral instructions.
🔒
session_startup.py SessionStart
SHA-256 integrity check on all 7 skills. Warns if tampered. Initializes session locks. Runs first — gate for everything else.
🏗
auto_prime_inject.py → PROJECT_CONTEXT.md L2 · auto ~2000 tok
Loads .claude/PROJECT_CONTEXT.md — full architecture overview: agents, commands, hooks, structure, key files. Git-hash validated. Stale if git hash changed → run /prime to refresh.
🗺
repo_map.py L2 · large repos only ~1000 tok
Only fires if project has ≥200 tracked files. Generates a TreeSitter symbol index ranked by importance — prevents blind file exploration. Skipped for small projects.
Why not auto-inject FACTS.md and MEMORY.md? These files are read on-demand — Claude reads them when starting project-specific work, not unconditionally every session. This saves ~2300 tokens on sessions where they're not needed (research tasks, new files, quick questions).
On-Demand Access

Memory Recall — When & How

Episodic memory is read when relevant, not burned at every session start. Claude decides based on task context.

FACTS.MD RECALL FLOW
💬
Task involves this project's code, configuration, or known patterns
Trigger: any task that requires project-specific knowledge (fixing a bug, adding a feature, running commands, modifying hooks).
📋
Read .claude/FACTS.md on-demand
Claude reads FACTS.md using the Read tool. Categories injected: CONFIRMED, GOTCHAS, PATHS, PATTERNS. STALE section excluded automatically by auto_fact_extractor.
✅
Trust hierarchy applied
CONFIRMED entries override inference. GOTCHAS checked before running commands. PATHS consulted before file exploration. PATTERNS used for known-good sequences.
MEMORY.MD RECALL FLOW
🔄
Resuming work after a gap (different session, continuing a feature)
Trigger: when context suggests continuing previous work — bug that was partially fixed, feature in progress, or unclear what changed recently.
📖
Read .claude/MEMORY.md on-demand
Claude reads MEMORY.md: rolling session history (last 30 entries max). Each entry: date, commit message, files changed, tasks completed. Git-diff based — ground truth not summaries.
🎯
Context oriented: what changed, what was fixed, what decisions were made
Use recent entries to understand project state. Entries are factual (git diff + commit) not subjective. Oldest entries auto-pruned after 30 sessions.
Automatic Writes

Memory Save — Fully Automatic

Zero manual effort. Hooks write to FACTS.md and MEMORY.md in the background after every relevant tool call and at session end.

⚡
After Every Bash Call
PostToolUse: Bash
  • auto_fact_extractor.py → FACTS.md
  • Success + known pattern → CONFIRMED
    Failure + error pattern → GOTCHA
    No LLM calls — pure pattern matching
📝
After Every Write
PostToolUse: Write
  • auto_fact_extractor.py → FACTS.md
  • Key path written → PATHS fact
    Tracks: .claude/, templates/, global-hooks/, docs/, scripts/, config files
🏁
Session End
Stop hooks
  • auto_memory_writer.py → MEMORY.md
  • validate_facts.py → prune FACTS.md
  • MEMORY: git diff + commit message.
    Only writes if something actually changed.
    FACTS: prunes entries older than 90 days.

When to create project-local agents/skills
✓ Create local when…
  • → Project has unique deploy / test workflow
  • → Domain expertise needed (payments, auth, ML)
  • → Need to override a global agent's behavior
  • → Project-specific code review rules
  • → Custom commands for this project's toolchain
✗ Don't create local when…
  • → Global agent already handles the task
  • → Behavior would be identical to global
  • → Task is one-off (just use the global)
  • → You'd want it in all projects (add to global-skills/ instead)
End-to-End

Full Request Lifecycle

From user message to tool execution to memory persistence — every touchpoint.

USER MESSAGE → TOOL CALLS → MEMORY PERSISTENCE
1️⃣
User sends message
UserPromptSubmit fires: kr_mode (Korean?), auto_prime_inject (force /prime if no cache), analyze_request (Caddy classifies: direct/orchestrate/rlm/fusion), auto_delegate (route if complex).
2️⃣
Claude reads episodic memory if task needs it on-demand
Project-specific task? → Read .claude/FACTS.md (verified facts, gotchas, known paths). Resuming work? → Read .claude/MEMORY.md (what changed in recent sessions). Both read using the Read tool — only consumed when needed.
3️⃣
Agents and skills selected
Claude resolves: local .claude/agents/ checked first → global ~/.claude/agents/ fallback. Same for skills. Caddy recommendation considered. Complex tasks → orchestrator spawns team. Simple tasks → direct execution.
4️⃣
PreToolUse gates every tool call
session_lock_manager: prevent concurrent file writes. unified-damage-control: block destructive patterns (rm -rf, force-push, etc.). auto_review_team: review risky Bash before execution.
5️⃣
Tool executes
Bash / Read / Write / Edit runs. Tool output lands in Layer 0 (sensory) — visible this turn only.
6️⃣
PostToolUse: memory + observability zero tokens
auto_fact_extractor: writes CONFIRMED/GOTCHA/PATHS facts to FACTS.md. auto_context_manager: compress cold tasks at 70% context. auto_error_analyzer: if Bash failed, inject diagnosis. auto_cost_warnings: budget alerts. All write to disk — no token cost.
7️⃣
Session ends → Stop hooks persist memory
auto_memory_writer: git diff → session summary written to .claude/MEMORY.md. validate_facts: prune 90-day stale facts. voice_done: audio notification. Memory files ready for next session.
