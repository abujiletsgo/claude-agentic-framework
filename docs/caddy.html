What is Caddy?

Automatic Task Classification

Caddy is a UserPromptSubmit hook that fires on every message you send. It classifies the request on multiple dimensions and injects a routing recommendation into Claude's context — all before Claude writes a single word.

What happens when you send a message
💬
You send a message
UserPromptSubmit hooks fire before Claude processes the message.
🔍
analyze_request.py — Caddy classifies the task
Keyword matching + lightweight Haiku call. Classifies on: complexity (simple/moderate/massive), quality (standard/critical), scope (known/moderate/unknown), domain (implement/research/debug/etc). Also suggests relevant skills.
📡
auto_delegate.py — injects routing recommendation
Based on classification, injects into context: recommended strategy, confidence score, relevant skills, and guidance note. You see this as the [Caddy] system-reminder.
🤖
Claude responds with appropriate strategy
Claude reads the Caddy recommendation and uses it to decide: respond directly, spawn a builder+validator pair, invoke the orchestrator, or use RLM for large-scale analysis.
The Caddy recommendation is advisory, not mandatory. Claude always has final judgment. If you explicitly ask for a specific approach, that overrides Caddy. But for ambiguous requests, Caddy consistently picks the right strategy.
Strategies

5 Execution Strategies

Caddy selects one of these strategies based on complexity, scope, and quality requirements.

direct fastest
Simple, known-scope tasks. 1-2 files, clear requirements, no exploration needed.
  • → Fix a typo or variable name
  • → Add a console.log for debugging
  • → Update a single config value
  • → Explain what a function does
team recommended most often
Multi-step tasks with moderate complexity. Builder (Sonnet) implements + Validator (Haiku) verifies in parallel.
  • → Add a new API endpoint with validation
  • → Fix a bug that spans 3-4 files
  • → Update documentation across a module
  • → Refactor a service class
orchestrate
Complex tasks requiring coordination across many files, unknown scope, or multiple specialist domains.
  • → Implement a full auth system
  • → Security audit of the codebase
  • → Performance investigation with unknown cause
  • → Cross-cutting refactor (e.g., add observability)
rlm
Massive codebase tasks where even a single agent would hit context limits. Recursive sub-agents explore slices.
  • → Full codebase audit (200+ files)
  • → Dependency analysis across a monorepo
  • → "Find all X across the entire project"
fusion best-of-N
Critical code where quality matters more than speed. Runs N parallel attempts, picks the best output.
  • → Security-critical cryptographic functions
  • → Payment processing logic
  • → Core algorithm implementations

Under the Hood

How Caddy Works Technically

Two hooks. One classifies, one recommends. Both fire before Claude reads your message.

analyze_request.py

UserPromptSubmit hook. Reads the prompt text, applies keyword matching for fast classification, then optionally calls Haiku for ambiguous cases.

Classification dimensions:

  • → complexity: simple / moderate / massive
  • → quality: standard / critical
  • → scope: known / moderate / unknown
  • → domain: implement / research / debug / document / review / explain

Config: data/caddy_config.yaml

auto_delegate.py

Second UserPromptSubmit hook. Reads the classification output from analyze_request and injects the recommendation into Claude's context as a system reminder.

What gets injected:

  • → strategy: direct / team / orchestrate / rlm / fusion
  • → confidence: 0–100%
  • → skills: relevant skill names
  • → guidance: one-line execution note

This is the [Caddy] line you see in session reminders.

Token cost: Caddy's analysis costs ~100 tokens per message (Haiku is cheap). The savings from routing correctly — avoiding an orchestrator spawn on a trivial task — far exceed this cost on any non-trivial session.
Customization

Tuning the Classifier

Caddy's keyword lists and thresholds are configurable. Edit data/caddy_config.yaml to add project-specific signals.

# data/caddy_config.yaml — example customization

complexity_signals:
  massive:
    - "entire codebase"
    - "full audit"
    - "migrate everything"
    - "your_project_specific_term"  # ← add your own
  simple:
    - "typo"
    - "comment"
    - "rename"

quality_signals:
  critical:
    - "payment"
    - "auth"
    - "crypto"
    - "pci"
    - "your_critical_module"  # ← add your critical paths

# Confidence thresholds
thresholds:
  direct_max_complexity: simple
  team_max_complexity: moderate
  orchestrate_min_complexity: massive
