Claude Agentic Framework

요청 라우팅

당신이 보내는 모든 메시지는 Claude가 응답하기 전에 분석됩니다. 요청 분류기인 Caddy가 최적의 전략을 결정하여, 불필요한 낭비 없이 적절한 깊이의 응답을 제공합니다.

Caddy란 무엇인가?

자동 태스크 분류

Caddy는 당신이 보내는 모든 메시지에 실행되는 UserPromptSubmit 훅입니다. 요청을 여러 차원으로 분류하고, Claude가 한 글자도 쓰기 전에 라우팅 권장사항을 컨텍스트에 주입합니다.

Claude가 메시지를 처리하기 전에 UserPromptSubmit 훅이 실행됩니다.

analyze_request.py — Caddy가 태스크를 분류합니다
키워드 매칭 + 경량 Haiku 호출. 복잡도(simple/moderate/massive), 품질(standard/critical), 범위(known/moderate/unknown), 도메인(implement/research/debug 등)을 분류합니다. 관련 스킬도 제안합니다.
📡
auto_delegate.py — 라우팅 권장사항을 주입합니다
분류 결과를 바탕으로 컨텍스트에 주입합니다: 권장 전략, 신뢰도 점수, 관련 스킬, 안내 메모. [Caddy] system-reminder로 표시됩니다.
🤖
Claude가 적절한 전략으로 응답합니다
Claude는 Caddy 권장사항을 읽고 직접 응답할지, 빌더+검증자 쌍을 생성할지, 오케스트레이터를 호출할지, 대규모 분석을 위해 RLM을 사용할지 결정합니다.
Caddy 권장사항은 참고용이지 강제가 아닙니다. Claude는 항상 최종 판단권을 가집니다. 특정 접근 방식을 명시적으로 요청하면 Caddy를 재정의합니다. 하지만 모호한 요청에서는 Caddy가 일관되게 올바른 전략을 선택합니다.
전략

5가지 실행 전략

Caddy는 복잡도, 범위, 품질 요구사항에 따라 다음 전략 중 하나를 선택합니다.

직접 (direct) 가장 빠름
간단하고 범위가 명확한 태스크. 파일 1-2개, 명확한 요구사항, 탐색 불필요.
  • → 오타 또는 변수명 수정
  • → 디버깅용 console.log 추가
  • → 단일 설정값 변경
  • → 함수 동작 설명
팀 (team) 가장 자주 권장됨
중간 복잡도의 다단계 태스크. 빌더(Sonnet)가 구현하고 검증자(Haiku)가 병렬로 검증합니다.
  • → 유효성 검사가 포함된 새 API 엔드포인트 추가
  • → 3-4개 파일에 걸친 버그 수정
  • → 모듈 전반의 문서 업데이트
  • → 서비스 클래스 리팩토링
오케스트레이트 (orchestrate)
여러 파일에 걸쳐 조율이 필요하거나, 범위가 불명확하거나, 여러 전문 도메인이 필요한 복잡한 태스크.
  • → 전체 인증 시스템 구현
  • → 코드베이스 보안 감사
  • → 원인 불명의 성능 문제 조사
  • → 횡단 관심사 리팩토링 (예: 옵저버빌리티 추가)
RLM
단일 에이전트도 컨텍스트 한계에 도달할 만큼 방대한 코드베이스 태스크. 재귀적 서브에이전트가 슬라이스 단위로 탐색합니다.
  • → 전체 코드베이스 감사 (200개 이상 파일)
  • → 모노레포 전체 의존성 분석
  • → "프로젝트 전체에서 X를 모두 찾아줘"
퓨전 (fusion) 최선 선택
속도보다 품질이 중요한 중요 코드. N개의 병렬 시도를 실행하고 최선의 결과물을 선택합니다.
  • → 보안이 중요한 암호화 함수
  • → 결제 처리 로직
  • → 핵심 알고리즘 구현
실제 분류 사례

프롬프트 분류 방식

실제로 입력할 내용과 Caddy가 권장하는 전략의 정확한 예시입니다.

메시지 내용 전략 이유
"README의 오타를 고쳐줘" 직접 (direct) 단일 파일, 사소한 변경, 위치 명확
"회원가입 폼에 입력 유효성 검사를 추가해줘" 팀 (team) 다단계: 폼 찾기, 유효성 추가, 테스트 — 범위는 명확
"로그인이 안 돼" 팀 (team) 디버그 + 수정 패턴, 중간 범위, 탐색 필요
"이중 인증을 추가해줘" 오케스트레이트 (orchestrate) 복잡한 기능, 여러 파일, 현재 인증 구조 불명확
"보안 감사를 해줄 수 있어?" 오케스트레이트 (orchestrate) 개방형, 여러 도메인, 높은 품질 요구
"이 코드베이스에서 에러를 처리하는 곳을 모두 찾아줘" RLM 횡단 관심사, 전체 코드베이스, 컨텍스트 한계 위험
"모범 사례를 적용해서 결제 청구 함수를 구현해줘" 퓨전 (fusion) 중요 코드 경로, 속도보다 품질 우선, 금융 로직
"useAuth 훅은 뭘 하는 거야?" 직접 (direct) 조사/설명 태스크, 파일 1-2개 읽기, 구현 없음
"전체 사용자 모듈을 의존성 주입 방식으로 리팩토링해줘" 오케스트레이트 (orchestrate) 대규모 범위, 파일 수 불명, 아키텍처 변경
분류 작동 원리

Caddy의 기술적 동작 방식

두 개의 훅. 하나는 분류하고, 하나는 권장합니다. 둘 다 Claude가 메시지를 읽기 전에 실행됩니다.

analyze_request.py

UserPromptSubmit 훅. 프롬프트 텍스트를 읽고, 빠른 분류를 위해 키워드 매칭을 적용한 후, 모호한 경우에는 선택적으로 Haiku를 호출합니다.

분류 차원:

  • → complexity(복잡도): simple / moderate / massive
  • → quality(품질): standard / critical
  • → scope(범위): known / moderate / unknown
  • → domain(도메인): implement / research / debug / document / review / explain

설정 파일: data/caddy_config.yaml

auto_delegate.py

두 번째 UserPromptSubmit 훅. analyze_request의 분류 결과를 읽고, 권장사항을 시스템 리마인더로 Claude의 컨텍스트에 주입합니다.

주입되는 내용:

  • → strategy(전략): direct / team / orchestrate / rlm / fusion
  • → confidence(신뢰도): 0–100%
  • → skills(스킬): 관련 스킬 이름
  • → guidance(안내): 한 줄 실행 메모

세션 리마인더에서 보이는 [Caddy] 줄이 바로 이것입니다.

토큰 비용: Caddy의 분석은 메시지당 약 100토큰이 소요됩니다 (Haiku는 저렴합니다). 올바른 라우팅으로 절약되는 비용 — 사소한 태스크에 오케스트레이터를 불필요하게 실행하지 않는 것 — 은 어떤 비사소한 세션에서도 이 비용을 훨씬 초과합니다.
Caddy 커스터마이징

분류기 튜닝

Caddy의 키워드 목록과 임계값은 설정 가능합니다. data/caddy_config.yaml을 편집하여 프로젝트별 신호를 추가하세요.

# data/caddy_config.yaml — 커스터마이징 예시

complexity_signals:
  massive:
    - "entire codebase"
    - "full audit"
    - "migrate everything"
    - "your_project_specific_term"  # ← 직접 추가
  simple:
    - "typo"
    - "comment"
    - "rename"

quality_signals:
  critical:
    - "payment"
    - "auth"
    - "crypto"
    - "pci"
    - "your_critical_module"  # ← 중요 경로 추가

# 신뢰도 임계값
thresholds:
  direct_max_complexity: simple
  team_max_complexity: moderate
  orchestrate_min_complexity: massive
