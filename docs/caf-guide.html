Section 1

Overview

CAF solves the coordination problem in AI-assisted engineering: how to safely delegate large tasks to fleets of Claude agents without losing control, burning tokens on the wrong model, or having agents clobber each other's work.

Multi-Model Cost Efficiency

Three-tier model selection (Opus → Sonnet → Haiku) delivers an estimated 50–60% cost savings vs. single-model deployments. Every task is routed to the cheapest model that can handle it.

48-Hook Observability

Safety, automation, and logging baked into every event: session startup, prompt submission, tool use, compaction, and shutdown. Hooks run in parallel and always exit 0.

Parallel Agent Execution

All agents in a wave launch in a single message. No serializing work that can run simultaneously. Minimum 3 agents for any non-trivial orchestration task.

Persistent Memory

SQLite FTS5 knowledge pipeline retains learnings across sessions. FACTS.md and MEMORY.md auto-update via PostToolUse and Stop hooks.

Full Autonomy + Guardrails

Yolo mode grants allow: ["*"] for zero-friction operation. Damage-control hooks with 100+ patterns and circuit breakers serve as the safety net.

Run Explorer Dashboard

Vue 3 + Bun web UI for visual replay of every orchestration run. Real-time WebSocket feed, session timeline, cost tracking, and side-by-side run comparison.

Repository Structure

One flat structure. All components installed from a single bash install.sh.

global-agents/   22 agent definition files (2 Opus, 15 Sonnet, 4 Haiku)

Section 2

Orchestration

The /orchestrate command is the entry point for all multi-agent work. The orchestrator agent spawns parallel teams in waves — it never does implementation work itself.

Complexity Sizing

The orchestrator sizes the task before spawning any agents. This determines which waves run and how many agents are allocated.

Complexity When to use Team size Waves
Trivial Rename, config value, copy edit Skip agents None
Simple Single-file bug, clear root cause 3 agents 0b → 1 → 2
Standard Multi-file feature, unclear scope 5–6 agents 0a → 0b → 1 → 2
Complex Cross-domain, 3+ subsystems 8+ agents 0a → 0b → 1 → 2

Wave Execution

Every wave launches all its agents in a single message. Never serialize agents that can run in parallel.

  1. 0

    Sizing + IPC Init (all runs)

    Generate orch_id via python3 -c "import time; print(f'orch_{int(time.time())}')". Create workspace at ~/.caf/orch/<orch_id>/ via bin/orch-shared init <orch_id>.

  2. 0a

    Consultation (Standard + Complex only)

    Domain consultants spawn in parallel — frontend-consultant, backend-consultant, architecture-consultant, security-consultant. Each produces a spec section. Output is synthesized into ~/.caf/orch/<id>/spec.md and user-approved before proceeding.

  3. 0b

    Research (parallel)

    One researcher per research question, all launched in one message. Results append to ~/.caf/orch/<id>/research.md. Researchers cover: patterns + prior art, security + attack surface, domain-specific questions.

  4. 1

    Build (parallel builders, minimum 3 total agents)

    Builders implement spec. Haiku for exact string/value replacements. Sonnet for logic, security, multi-file interdependencies. Each builder writes a log to results/<name>.md.

  5. 2

    QA Loop

    Validator (Haiku) runs first — reads spec, exercises behavior, writes PASS/FAIL to qa-report.md. On PASS: critical-analyst (Sonnet) reviews for edge cases. On FAIL: debugger → fresh builders → re-validate. After 2 failures: escalate to user.

Advisor Tool (advisor_20260301)

The advisor tool is an official Anthropic beta feature (launched April 9, 2026). A faster executor model (Sonnet or Haiku) runs the task end-to-end and autonomously calls Opus mid-generation when it hits a decision fork — all within a single API request, no extra round trips.

Benchmarks: Sonnet + Opus advisor → +2.7 pts SWE-bench, −11.9% cost. Haiku + Opus advisor → +109% BrowseComp (19.7% → 41.2%). The executor handles routine token generation; Opus only fires at the moments it changes outcomes.

How it actually works

The executor model calls advisor() as a tool. Anthropic's server runs a separate Opus inference pass on the full conversation transcript and returns plaintext guidance. The executor continues with that guidance — no NEED_ADVISOR text markers, no orchestrator scanning, no separate agent. It is a server-side tool inside one API call.

# Messages API (Python SDK) — requires API key + beta access
response = client.beta.messages.create(
    model="claude-sonnet-4-6",          # executor
    betas=["advisor-tool-2026-03-01"],
    tools=[{
        "type": "advisor_20260301",
        "name": "advisor",
        "model": "claude-opus-4-6",     # advisor
        "max_uses": 5,
    }],
    messages=[{"role": "user", "content": task}],
)

When the executor calls advisor()

The model decides autonomously. The built-in tool description nudges it toward these moments:

WhenExample
Before substantive work (after orientation reads)Explored the repo, now about to write the first file
Before declaring doneAfter writing + running tests — commit or fix?
When stuck / not convergingSame fix failing three times
When reconsidering approachCurrent approach works but feels wrong

CAF implementation: bin/advisor

For API key users, run any task through Sonnet + Opus advisor:

# Sonnet executor + Opus advisor (default)
bin/advisor "Implement a rate limiter with token bucket algorithm in Go"

# Haiku executor + Opus advisor (cheapest, good for mechanical tasks)
bin/advisor --model haiku "Format this JSON schema to match the v2 spec"

# Pipe from stdin
echo "Task description" | bin/advisor

The script prints the assistant output to stdout and a cost breakdown (per iteration, with executor vs advisor token split) to stderr. Requires ANTHROPIC_API_KEY set to a real API key from console.anthropic.com.

Beta header in settings

templates/settings.json.template includes advisor-tool-2026-03-01 in ANTHROPIC_CUSTOM_HEADERS alongside the existing cache TTL header. This sends the beta header on all Claude Code API calls — once Anthropic enables full native support for OAuth users, it will take effect without any further config change.

The orchestrator selects the cheapest model that can correctly complete each build task.

Hard Rules

  • ! Consultants first. No building without approved spec. Skip Wave 0a only for Simple tasks.
  • ! Minimum 3 agents for any non-trivial task.
  • ! All agents in a wave launch in ONE message. Never serialize work that can run in parallel.
  • ! Write events at every wave boundary. Run Explorer reads these to reconstruct the timeline.
  • ! Escalate after 2 QA failures. Never loop forever. Bring the user back in.
  • ! The orchestrator never does implementation work itself. No Read/Edit/Grep on implementation files.

Section 3

Session Recording

Every Claude session is recorded as a .jsonl file — one JSON object per line, append-only. The Rust hook is 6–32× faster than the Python equivalent and always exits 0, never blocking Claude.

Implementation

caf-hooks/src/hooks/session_recorder.rs — compiled into the caf-hooks Rust binary.

Storage

~/.caf/sessions/<session_id>.jsonl — one file per session, append-only JSONL format.

Reader

apps/run-explorer/server/src/services/sessionParser.ts — parses JSONL into typed session objects.

Event Fields by Type

Three Claude hook events are captured. Each line in the JSONL file corresponds to one event.

Event Fields recorded Notes
SessionStart ts, ms, type, cwd, orch_run_id orch_run_id only included if ~/.caf/current_orch_id exists and is < 4 hours old
UserPromptSubmit ts, ms, type, cwd, prompt prompt truncated to 500 chars to control disk usage
Stop ts, ms, type, cwd Marks end of session; used to compute durationSeconds

All timestamps are RFC 3339 with millisecond precision. The ms field is Unix epoch milliseconds for duration calculation.

Orch Run Correlation

Sessions spawned during an orchestration run are linked back to that run via orch_run_id.

orch-shared init <id>
→
Writes ~/.caf/current_orch_id
{orch_id, started_at}
→
SessionStart hook reads file
→
Injects orch_run_id into JSONL
→
Run Explorer joins sessions to run
Staleness check: If current_orch_id is more than 4 hours (14,400 seconds) old, the session recorder ignores it and omits orch_run_id. This prevents sessions from being incorrectly linked to stale runs after a crash or long pause.

JSONL Format Example

// ~/.caf/sessions/abc123.jsonl
{"ts":"2026-04-14T09:00:00.000Z","ms":1744621200000,"type":"SessionStart","cwd":"/Users/tom/caf-team","orch_run_id":"orch_1776158991"}
{"ts":"2026-04-14T09:00:05.123Z","ms":1744621205123,"type":"UserPromptSubmit","cwd":"/Users/tom/caf-team","prompt":"/orchestrate add dark mode to the dashboard"}
{"ts":"2026-04-14T09:04:30.456Z","ms":1744621470456,"type":"Stop","cwd":"/Users/tom/caf-team"}

Subsystems

Subsystem Purpose
Damage Control 100+ patterns block or confirm destructive Bash commands. Path protection: zero-access, read-only, no-delete zones. Runs on every PreToolUse.
Caddy Classifier Classifies every user prompt on 4 dimensions: direct / orchestrate / rlm / fusion. Auto-routes complex tasks to the right strategy.
Knowledge Pipeline SQLite FTS5 persistent cross-session memory. Facts extracted on every PostToolUse; MEMORY.md updated on every Stop.
RepoMap TreeSitter symbol index built on SessionStart for repos with ≥ 200 source files. Provides fast symbol lookup without re-reading files.
Circuit Breakers Prevent runaway hooks from cascading failures. State stored in circuit_breakers/. Auto-recovers after 60 seconds. Delete state file to reset immediately.
Session Management Init, file conflict detection, session lock management, cleanup on stop.

Hook Pipeline (simplified)

User prompt
  → [Caddy: classify task → direct / orchestrate / rlm / fusion]
  → Claude processes
  → [PreToolUse: damage control + lock manager + orchestrator guard]
  → Tool executes
  → [PostToolUse: logging + error analysis + cost warnings + fact extraction]
  → Session ends
  → [Stop: memory writer + fact validator + session recorder]

Next session starts:
  → [SessionStart: inject past learnings + RepoMap for large repos]

Execution Model

RuntimeHow it runsSpeed
Python uv run --script <hook>.py with inline # /// script dependencies header Baseline
Rust caf-hooks binary — compiled with cargo build --release, copied to ~/.local/bin/ 6–32× faster
Circuit breaker reset: If hooks are erroring everywhere, check ~/.claude/circuit_breakers/. Delete the relevant state file or wait 60 seconds for auto-recovery.

Install Steps

  1. 0
    Check and install prerequisites uv, python3, git, officecli, claude — auto-installed if missing
  2. 0b
    Create directories data/, orch/, orch_state/, health/, /tmp/claude/
  3. 0b2
    Symlink orch-shared bin/orch-shared → ~/.local/bin/orch-shared for global PATH access
  4. 0b3
    Symlink shared libraries Shared libs → lib/ in claude dir
  5. 0c
    Pre-warm uv dependency cache pyyaml, pydantic, anthropic — eliminates first-run latency
  6. 0c
    Build Rust binary (if cargo available) cd caf-hooks && cargo build --release → copied to ~/.local/bin/caf-hooks
  7. 1
    Validate all hook .py files exist Checks every file referenced in the template before generating settings
  8. 2
    Generate settings.json from template Injects absolute PATH and CAF_HOOKS_DIR into the env block
  9. 3
    Symlink 16 commands global-commands/ → commands dir in claude config
  10. 4
    Symlink 30 skills global-skills/ → skills dir in claude config
  11. 5
    Symlink 22 agents global-agents/ → agents dir in claude config
  12. 6
    Generate docs uv run scripts/generate_docs.py
  13. 7
    Append Yolo block to global CLAUDE.md Grants allow: ["*"] — skipped if already present
  14. 8
    Install git pre-push hook Auto-regenerates docs before every push
  15. 9
    Init FACTS.md and MEMORY.md Created with empty scaffolding if not present
  16. 10
    Hook smoke test Verifies uv, pyyaml, and pydantic are functional
  17. 10b
    Doctor checks Orch dir writable, orch-shared accessible, symlink counts match expected
  18. 11
    Final summary Prints install report and any warnings
To uninstall: run ./uninstall.sh. This removes symlinks and the generated settings.json but does not delete your data or memory files.

Section 9

Troubleshooting

Most CAF issues fall into one of three categories: hook circuit breakers, settings drift, or missing symlinks. The table below covers every common symptom.

Symptom Cause Fix
Hook errors on every tool use Circuit breaker tripped Delete state file from ~/.claude/circuit_breakers/ or wait 60 seconds for auto-recovery
Settings mismatch after editing hooks Edited settings.json directly Edit templates/settings.json.template, then run bash install.sh
Hook file missing error in Claude Deleted a hook file that settings.json references Stub first (empty script that exits 0), reinstall, then delete
Hooks stop working after directory move Moved framework directory without updating paths Update paths in templates/settings.json.template first, then move, then reinstall
pip install errors in hook output Hook uses pip instead of uv Replace pip install with uv run in the hook script
orch-shared: command not found Symlink not in PATH Check ~/.local/bin/orch-shared exists and ~/.local/bin is in your PATH
Session not linked to orchestration run current_orch_id is stale (> 4 hours old) Restart the orchestration run; orch-shared init will refresh the file
Install fails on first hook run Stale uv dependency cache Run uv cache clean && bash install.sh
Caddy classifier always returns direct Caddy config not loaded or caddy_config.yaml missing Verify data/caddy_config.yaml exists; run bash install.sh to re-generate paths
Run Explorer shows no sessions API server not running or sessions dir empty Ensure bun run dev is running in apps/run-explorer/; check ~/.caf/sessions/

Context Engineering Tips

Memory Tier File When loaded Contents
Tier 1 PROJECT_CONTEXT.md Auto at session start Authoritative project snapshot — injected by session_startup.py
Tier 2 FACTS.md On-demand Verified facts: CONFIRMED > GOTCHAS > PATHS > PATTERNS
Tier 3 MEMORY.md On-demand Session summaries, git-diff-based, max 30 entries
Epistemic discipline: Agent outputs must label claims as OBSERVED (cite file:line), INFERRED (state reasoning chain), or SPECULATIVE (flag explicitly). Never present inference as observation.
