Global Agents

The 8 Framework Agents

Installed globally at ~/.claude/agents/ via bash install.sh. Available in every project without any per-project setup.

orchestrator Opus global
Primary multi-agent coordinator. Analyzes complex tasks, selects optimal execution strategy, and coordinates specialized teams. The entry point for anything involving 4+ files or 5+ steps.
  • OAuth2 implementation from scratch
  • Security audit of the full codebase
  • Large-scale refactoring across multiple modules
  • Performance investigation with unknown root cause
Invoke
/orchestrate "implement user authentication with JWT"
Orchestrator analyzes → spawns project-architect (design) + researcher (explore) + builder (implement) in parallel
project-architect Opus global
Architecture oversight and design decisions. Reviews structural choices, identifies patterns, flags anti-patterns, and proposes system-level improvements. Uses Opus for the reasoning depth architecture requires.
  • Reviewing a proposed data model before building
  • Deciding between microservices vs monolith
  • Designing the API surface for a new feature
  • Identifying coupling and dependency issues
Invoke via orchestrator or Task tool
Task: project-architect, "review the current auth module structure"
Returns: 2-3 sentence architectural assessment with specific concerns and recommendations
critical-analyst Opus global
Questions everything. Used during planning and decision-making to stress-test assumptions, identify blind spots, and challenge proposed approaches. Intentionally contrarian to surface problems early.
  • Reviewing an implementation plan before starting
  • Validating that a "fix" actually solves the problem
  • Checking security assumptions in new code
  • Pre-mortem analysis on a risky deployment
Example output style
Why are you storing JWTs in localStorage? That exposes you to XSS.
This migration assumes zero downtime but the table has no index on the WHERE clause.
researcher Sonnet global
Deep codebase exploration specialist. Reads many files, searches patterns, synthesizes findings. Designed to run in isolation — it does the heavy reading so your primary context doesn't fill up.
  • Finding all usages of a deprecated function
  • Understanding how an unfamiliar module works
  • Mapping data flow across 10+ files
  • Answering "where does X happen in this codebase?"
Invoke
/research "how does the session management work in this app?"
Agent reads auth/, middleware/, session files → returns 2-3 sentence finding with key file paths
meta-agent Sonnet global
Generates new Claude Code agent configuration files. Give it a description of what you need and it produces a complete, ready-to-use agent definition. Uses project context to make agents project-aware.
  • Creating a domain-specialist agent for your project
  • Generating a custom reviewer with project-specific rules
  • Building a deployment agent tailored to your stack
Example
"create an agent specialized in our GraphQL schema validation"
Produces .claude/agents/graphql-validator.md ready to use
scout-report-suggest Sonnet global
Targeted analysis and reporting. Read-only specialist — scouts for issues, identifies problem locations, and suggests resolutions without making changes. Returns concise reports.
  • Finding all places a bug could originate
  • Scanning for security anti-patterns in a module
  • Identifying performance bottlenecks before optimizing
  • Pre-check before a refactoring session
Output format
Problem: N+1 query in UserService.getWithPosts() (line 47)
Location: src/services/user.service.ts:47
Suggestion: Use include: { posts: true } in the Prisma query
rlm-root Sonnet global
Recursive Language Model controller. Handles infinite-scale codebase analysis by spawning child agents recursively. Each agent reads a slice, summarizes, and reports up. No context rot regardless of codebase size.
  • Full codebase audit (500+ files)
  • Cross-cutting concern analysis (e.g., "find all error handling")
  • Dependency graph construction for large monorepos
Invoke
/rlm "find all places we handle authentication across the codebase"
Spawns recursive sub-agents, each reads a subtree → consolidated report with no single agent hitting context limits
docs-scraper Haiku global
Documentation fetching specialist. Fast and cheap — uses Haiku for speed. Scrapes URLs, converts to markdown, saves structured output. Best for bulk documentation tasks.
  • Fetching API documentation for a library
  • Saving reference docs locally for offline use
  • Bulk-scraping changelog pages
Example
fetch and save the Prisma query API documentation
Scrapes docs, saves to docs/prisma-query-api.md with proper formatting
Global vs Local

Layered Agent Architecture

Global agents are always available. Local agents live in your project and override globals with the same name — letting you customize behavior per project without touching the framework.

🌐 Global Agents

~/.claude/agents/

Installed by bash install.sh. Available in every Claude Code session, every project, every directory.

Best for: General-purpose tasks that apply across projects — research, orchestration, architecture review.

📁 Local Agents

.claude/agents/

Project-scoped. Claude Code loads these first — if a local agent has the same name as a global one, local wins.

Best for: Domain expertise, project-specific workflows, overriding global agent behavior.
Resolution order — Claude Code agent lookup
📁
.claude/agents/ checked first
If a matching agent exists locally, it's used — regardless of what's global. Local always overrides. This is how you customize orchestrator behavior for a specific project without touching the global file.
🌐
~/.claude/agents/ fallback
If no local agent matches, the global is used. No config needed — it just works.
Custom Agents

Creating a Local Agent

Any .md file in .claude/agents/ becomes an agent. Claude Code auto-discovers them — no install, no restart needed.

---
name: payments-specialist
description: >
  Expert in Stripe integration, webhook handling, and PCI compliance
  for this project. Use for any payment-related code, error handling,
  or security review of billing flows.
model: claude-sonnet-4-6
tools:
  - Read
  - Glob
  - Grep
  - Bash
---

# Payments Specialist

You are a payments domain expert for this project. You know:
- Our Stripe integration is in `src/payments/` using Stripe Node SDK v14
- Webhooks are handled in `src/api/webhooks/stripe.ts`
- Test mode keys are in `.env.test`, never in source
- All payment amounts are stored in cents (integer), never floats

## Rules
- Always validate webhook signatures before processing
- Never log full card numbers or CVV — last 4 digits only
- Amount validation: check for negative values and overflow
- Use idempotency keys on all charge attempts
Pro tip: Include your project's actual file paths, conventions, and gotchas in the agent definition. An agent that knows src/payments/stripe.ts exists is far more useful than a generic payments expert.
Override example: Create .claude/agents/orchestrator.md to give the orchestrator project-specific routing rules — e.g., always use the payments-specialist for anything in src/payments/.
Spawning Agents

How Agents Call Each Other

Agents are spawned using the Task tool. They run in isolation — their context is separate from the parent, preventing context rot on long tasks.

Example: orchestrator spawning a team
💬
User: "add two-factor authentication"
Caddy classifies as: complex implement, scope: unknown. Routes to orchestrator.
🎯
orchestrator (Opus) — analyzes and plans
Reads PROJECT_CONTEXT.md, FACTS.md, identifies existing auth code. Decides: need to explore current auth, design 2FA flow, implement, validate.
⚡
Spawns 2 agents in parallel
researcher (Sonnet): "explore current auth module, return: file paths, current session strategy, existing middleware"
project-architect (Opus): "design 2FA integration points given our current auth approach"
📋
Synthesizes findings → implements
Both agents return 2-3 sentence findings. Orchestrator combines them into an implementation plan, then either executes directly or spawns a builder agent.
Key rule: Sub-agents return findings, not raw code dumps. "Auth is in src/auth/session.ts using express-session with Redis" — not 200 lines of file contents. Keeps the parent context clean.
