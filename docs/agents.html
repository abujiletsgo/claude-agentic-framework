Claude Agentic Framework

에이전트 시스템

3개의 모델 티어에 걸친 8개의 전문 에이전트. 각 에이전트는 특정 유형의 사고를 위해 목적에 맞게 설계되었습니다 — 심층 아키텍처 검토부터 가믈운 문서화 작업까지.

Global Agents

8가지 기본 에이전트

bash install.sh를 통해 ~/.claude/agents/에 전역으로 설치됩니다. 프로젝트별 설정 없이 모든 프로젝트에서 사용 가능합니다.

orchestrator Opus global
기본 멀티 에이전트 코디네이터. 복잡한 작업을 분석하고, 최적의 실행 전략을 선택하며, 전문 팀을 조율합니다. 4개 이상의 파일이나 5단계 이상의 작업이 포함된 모든 것의 진입점입니다.
  • OAuth2 첫점부터 구현
  • 전체 코드베이스 보안 감사
  • 여러 모듈에 걸친 대규모 리팩토링
  • 근본 원인을 알 수 없는 성능 조사
호출
/orchestrate "implement user authentication with JWT"
Orchestrator 분석 → project-architect(설계) + researcher(탐색) + builder(구현)을 병렬로 생성
project-architect Opus global
아키텍처 감독 상 설계 결정 담당. 구조적 선택을 검토하고, 패턴을 식별하며, 안티패턴을 지적하고, 시스템 수준의 개선안을 제안합니다. 아키텍처에 필요한 추론 깊이를 위해 Opus를 사용합니다.
  • 구현 전 제안된 데이터 모델 검토
  • 마이크로서비스 vs 모놀리스 결정
  • 새 기능의 API 인터페이스 설계
  • 결합도와 의존성 문제 식별
orchestrator 또는 Task 도구를 통해 호출
Task: project-architect, "review the current auth module structure"
반환: 구체적인 우려 사항과 권고안이 포함된 2-3문장의 아키텍처 평가
critical-analyst Opus global
모든 것에 의문을 제기합니다. 계획 상 의사결정 단계에서 가정을 검증하고, 맹점을 파악하며, 제안된 접근 방식에 이의를 제기하는 데 사용됩니다. 문제를 조기에 발견하기 위해 의도적으로 반론을 제시합니다.
  • 시작 전 구현 계획 검토
  • "수정 사항"이 실제로 문제를 해결하는지 검증
  • 새 코드의 보안 가정 확인
  • 위험한 배포에 대한 사전 분석
출력 스타일 예시
왜 JWT를 localStorage에 저장하나요? XSS에 노출됩니다.
이 마이그레이션은 무중단을 가정하지만 WHERE 절에 인덱스가 없습니다.
researcher Sonnet global
코드베이스 심층 탐색 전문가. 많은 파일을 읽고, 패턴을 검색하며, 결과를 종합합니다. 격리된 환경에서 실행되도록 설계되어 — 무거운 읽기 작업을 수행하으므로 기본 콘텍스트가 가득 차지 않습니다.
  • 더 이상 사용되지 않는 함수의 모든 사용처 찾기
  • 낙선 모듈의 동작 방식 이해
  • 10개 이상의 파일에 걸친 데이터 흐름 매핑
  • "이 코드베이스에서 X는 어디서 발생하나요?" 답변
호출
/research "how does the session management work in this app?"
에이전트가 auth/, middleware/, session 파일 읽기 → 핵심 파일 경로가 포함된 2-3문장 결과 반환
meta-agent Sonnet global
새로운 Claude Code 에이전트 설정 파일을 생성합니다. 필요한 내용을 설명하면 완전하고 바로 사용 가능한 에이전트 정의를 생성합니다. 프로젝트 콘텍스트를 활용하여 프로젝트에 특화된 에이전트를 만듭니다.
  • 프로젝트용 도메인 전문 에이전트 생성
  • 프로젝트별 규칙이 있는 커스텀 리빰어 생성
  • 기술 스택에 맞춰 맞는 배포 에이전트 구축
예시
"create an agent specialized in our GraphQL schema validation"
바로 사용 가능한 .claude/agents/graphql-validator.md 생성
scout-report-suggest Sonnet global
타겟 분석 상 보고 전문가. 읽기 전용 전문가 — 문제를 스카우팅하고, 문제 위치를 파악하며, 변경 없이 해결첸을 제안합니다. 간결한 보고서를 반환합니다.
  • 버그가 발생할 수 있는 모든 위치 찾기
  • 모듈의 보안 안티패턴 스캔
  • 최적화 전 성능 병목 지점 파악
  • 리팩토링 세션 전 사전 점검
출력 형식
문제: UserService.getWithPosts()의 N+1 쿼리 (47번째 줄)
위치: src/services/user.service.ts:47
제안: Prisma 쿼리에 include: { posts: true } 사용
rlm-root Sonnet global
재귀적 언어 모델 콘트롤러. 자식 에이전트를 재귀적으로 생성하여 무한 규모의 코드베이스 분석을 처리합니다. 각 에이전트는 일부를 읽고, 요약하여, 상위로 보고합니다. 코드베이스 크기에 관계없이 콘텍스트 오염이 없습니다.
  • 전체 코드베이스 감사 (500개 이상의 파일)
  • 횟단 관심사 분석 (예: "모든 오류 처리 찾기")
  • 대형 모노레포의 의존성 그래프 구성
호출
/rlm "find all places we handle authentication across the codebase"
재귀적 서브 에이전트 생성, 각각 하위 트리 읽기 → 단일 에이전트가 콘텍스트 한계에 도달하지 않는 통합 보고서
docs-scraper Haiku global
문서 수집 전문가. 빠르고 저렴합니다 — 속도를 위해 Haiku를 사용합니다. URL을 스크랙하고, 마크다운으로 변환하며, 구조화된 출력을 저장합니다. 대량 문서화 작업에 적합합니다.
  • 라이브러리의 API 문서 가져오기
  • 오프라인 사용을 위해 참조 문서 로컬에 저장
  • 변경 이력 페이지 대량 스크랙
예시
fetch and save the Prisma query API documentation
문서 스크랙, 적절한 형식으로 docs/prisma-query-api.md에 저장
RLM: Recursive Language Model

RLM이란 무엇인가?

RLM(Recursive Language Model)은 에이전트가 코드베이스 전체를 컨텍스트에 적재하지 않고도 무한 규모의 코드베이스를 분석할 수 있는 아키텍처 패턴입니다. 컨텍스트 오염(Context Rot) 문제를 근본적으로 해결합니다.

❌ 전통적 방식의 문제

에이전트가 파일을 직접 읽어 컨텍스트에 쌓습니다:

read("auth.py")       # 500줄
read("session.py")    # 300줄
read("middleware.py") # 400줄
# 컨텍스트: 1,200줄 원시 코드
# 주의력: 40% → 버그 놓침

✅ RLM 방식

Root Agent는 탐색만 하고, 분석은 서브에이전트에 위임합니다:

search("auth|login")       # 위치 파악
peek("auth.py", 30줄)      # 방향 확인
delegate(sub_agent, slice) # 분석 위임
# 컨텍스트: ~5k 토큰
# 주의력: 99% → 버그 발견
RLM 실행 루프 — 4단계
1
Orient — 구조 파악
Glob으로 파일 목록 확인, README를 30줄만 읽어 코드베이스 구조 파악. 아무것도 분석하지 않습니다 — 지도만 그립니다.
2
Target — 타겟 특정
Grep으로 관련 패턴 검색 → 파일 경로 + 라인 번호 확보. 관련 섹션을 30-50줄씩만 peek합니다.
3
Delegate — 병렬 위임 ← 핵심
각 타겟 섹션을 독립적인 서브에이전트에게 위임합니다. 서브에이전트는 해당 슬라이스만 보고 2-3문장 요약을 반환합니다. 모든 위임은 단일 메시지로 병렬 실행됩니다.
4
Synthesize — 요약 종합
Root Agent는 서브에이전트 요약만 수집해 최종 답변을 도출합니다. Root 컨텍스트: 항상 ~5k 토큰 유지. 필요시 1단계로 돌아가 반복합니다.

왜 더 저렴한가?

전통 방식은 200k 토큰을 모두 로드합니다. RLM은 검색(무료) + 25개 서브에이전트(각 2k 토큰) = 총 50k 토큰. 75% 비용 절감, 주의력은 99% 유지.

언제 사용하나?

  • 코드베이스 전체 보안 감사
  • 500개 이상 파일 분석
  • 횡단 관심사 탐색
  • 모노레포 의존성 그래프 구성
핵심 규칙: Root Agent는 원시 데이터를 절대 직접 분석하지 않습니다. 오직 탐색 코드(Grep/Glob/Read 최대 50줄)와 서브에이전트 요약만 컨텍스트에 유지합니다.

/rlm "find all authentication-related code across the codebase"
/rlm "audit entire codebase for SQL injection vulnerabilities"

전체 RLM 아키텍처 문서: guides/RLM_ARCHITECTURE.md

Global vs Local

글로벌 vs 로컬 에이전트

글로벌 에이전트는 항상 사용 가능합니다. 로컬 에이전트는 프로젝트 내에 위치하며, 같은 이름의 글로벌 에이전트를 재정의합니다 — 프레임워크를 건드리지 않고 프로젝트별로 동작을 커스터마이징할 수 있습니다.

🌐 글로벌 에이전트

~/.claude/agents/

bash install.sh로 설치됩니다. 모든 Claude Code 세션, 모든 프로젝트, 모든 디렉토리에서 사용 가능합니다.

적합한 용도: 프로젝트 전반에 걸쳐 적용되는 범용 작업 — 리서치, 오케스트레이션, 아키텍처 검토.

📁 로컬 에이전트

.claude/agents/

프로젝트 범위. Claude Code는 이를 먼저 로드합니다 — 로컬 에이전트가 글로벌 에이전트와 같은 이름을 가지면, 로컬이 우선합니다.

적합한 용도: 도메인 전문 지식, 프로젝트별 워크플로우, 글로벌 에이전트 동작 재정의.
해석 순서 — Claude Code 에이전트 조회
📁
.claude/agents/ 먼저 확인
로컬에 일치하는 에이전트가 있으면 글로벌과 관계없이 해당 에이전트가 사용됩니다. 로컬이 항상 우선합니다. 이를 통해 글로벌 파일을 건드리지 않고 특정 프로젝트의 orchestrator 동작을 커스터마이징할 수 있습니다.
🌐
~/.claude/agents/ 폴백
로컬 에이전트가 없으면 글로벌이 사용됩니다. 별도 설정 없이 — 쟜준 작동합니다.
Custom Agents

로컬 에이전트 만들기

.claude/agents/의 모든 .md 파일은 에이전트가 됩니다. Claude Code가 자동으로 인식합니다 — 설치나 재시작이 필요 없습니다.

---
name: payments-specialist
description: >
  Expert in Stripe integration, webhook handling, and PCI compliance
  for this project. Use for any payment-related code, error handling,
  or security review of billing flows.
model: claude-sonnet-4-6
tools:
  - Read
  - Glob
  - Grep
  - Bash
---

# Payments Specialist

You are a payments domain expert for this project. You know:
- Our Stripe integration is in `src/payments/` using Stripe Node SDK v14
- Webhooks are handled in `src/api/webhooks/stripe.ts`
- Test mode keys are in `.env.test`, never in source
- All payment amounts are stored in cents (integer), never floats

## Rules
- Always validate webhook signatures before processing
- Never log full card numbers or CVV — last 4 digits only
- Amount validation: check for negative values and overflow
- Use idempotency keys on all charge attempts
Pro tip: 에이전트 정의에 프로젝트의 실제 파일 경로, 콘비션, 주의 사항을 포함하세요. src/payments/stripe.ts가 존재한다는 것을 아는 에이전트는 일반적인 결제 전문가보다 훨씬 유용합니다.
재정의 예시: .claude/agents/orchestrator.md를 생성하여 orchestrator에 프로젝트별 라우팅 규칙을 부여합니다 — 예: src/payments/의 모든 것에 대해 항상 payments-specialist를 사용.
Spawning Agents

에이전트 호출 방법

에이전트는 Task 도구를 사용하여 생성됩니다. 격리된 환경에서 실행되며 — 콘텍스트가 부모와 분리되어 오랜 작업에서도 콘텍스트 오염을 방지합니다.

예시: orchestrator가 팀을 생성하는 과정
💬
사용자: "add two-factor authentication"
Caddy가 분류: 복잡한 구현, 범위: 미확인. orchestrator로 라우팅.
🎯
orchestrator (Opus) — 분석 상 계획
PROJECT_CONTEXT.md, FACTS.md 읽기, 기존 인증 코드 파악. 결정: 현재 인증 탐색, 2FA 흐름 설계, 구현, 검증 필요.
⚡
2개의 에이전트를 병렬로 생성
researcher (Sonnet): "현재 인증 모듈 탐색, 반환: 파일 경로, 현재 세션 전략, 기존 미들웨어"
project-architect (Opus): "현재 인증 방식을 고려한 2FA 통합 지점 설계"
📋
결과 종합 → 구현
두 에이전트 모두 2-3문장의 결과를 반환합니다. Orchestrator가 이를 구현 계획으로 통합한 후 직접 실행하거나 builder 에이전트를 생성합니다.
핵심 규칙: 서브 에이전트는 원시 코드 덤프가 아닌 결과를 반환합니다. "인증은 express-session과 Redis를 사용하는 src/auth/session.ts에 있습니다" — 200줄의 파일 내용이 아닙니다. 부모 콘텍스트를 깔끔하게 유지합니다.
